# Personal Agent Phase 4.5 Design

## Goal

Phase 4.5 upgrades the current memory layer from prompt-level correction to explicit memory stratification. The target is a lightweight design inspired by `akashic-agent`, without introducing a vector store, transactional memory writes, or a full optimizer pipeline.

This phase adds two primary capabilities:

- A dedicated `SELF.md` self-model that is injected into system context separately from user memory
- A structured `RECENT_CONTEXT.md` generated by batch consolidation instead of rolling raw user-message appends

The design also introduces minimal consolidation metadata so recent-context generation becomes incremental and restart-safe.

## Scope

Included in this phase:

- Add `SELF.md` to the memory workspace
- Add `consolidation_state.json` to track recent-context consolidation progress
- Change `RECENT_CONTEXT.md` to a fixed `Compression + Recent Turns` structure
- Trigger consolidation every 3 new user messages
- Keep `MEMORY.md`, `PENDING.md`, and `HISTORY.md` with clarified roles
- Allow dynamic `SELF.md` updates only through a restricted schema

Explicitly excluded from this phase:

- Vector retrieval or semantic search
- Transaction logs, snapshots, rollback, or two-phase commit
- A background optimizer process
- LLM-powered summarization for consolidation
- Injecting `HISTORY.md` or `PENDING.md` directly into prompt context

## Problem Statement

The current implementation improves continuity mainly by changing prompt wording and injecting `MEMORY.md` with a rolling `RECENT_CONTEXT.md`. That suppresses obvious continuity failures, but it still relies on the model to reinterpret raw recent user lines correctly.

The current weak points are:

- `RECENT_CONTEXT.md` is built as a rolling list of raw user lines rather than a structured summary
- The system has no explicit notion of which history has already been consolidated
- `SELF` information is embedded in the main system prompt instead of living as a separate self-model
- Assistant self-descriptions are suppressed during replay, but there is no stronger structural separation between self-model, user memory, and raw history

Phase 4.5 addresses those issues by separating storage roles and introducing a bounded consolidation pipeline.

## Architecture

The memory layer is split into six persistent artifacts:

- `SELF.md`
  The agent self-model. Injected into system context.
- `MEMORY.md`
  Stable long-term user facts. Injected into system context.
- `RECENT_CONTEXT.md`
  Structured recent summary. Injected into system context.
- `PENDING.md`
  Staging area for candidate stable facts and candidate self-style updates. Not injected into prompt context.
- `HISTORY.md`
  Raw conversation event log used as consolidation input. Not injected into prompt context.
- `consolidation_state.json`
  Consolidation cursor and counters. Not injected into prompt context.

### Prompt Injection Order

Prompt assembly should become:

1. Main system prompt
2. `SELF.md`
3. `MEMORY.md`
4. `RECENT_CONTEXT.md`
5. Recent replayed conversation turns
6. Current user message

`PENDING.md` and `HISTORY.md` remain retrieval and processing inputs only.

## File Responsibilities

### `SELF.md`

`SELF.md` contains the agent self-model and is kept separate from user facts.

It uses a fixed markdown schema:

```md
## Identity
- Name: Jarvis
- Role: Personal Telegram agent

## Capabilities
- Can answer user questions with the configured LLM
- Can use trusted memory files managed by the application
- Can maintain continuity through the app-managed memory layers

## Interaction Style
- Prefer concise, direct responses
- Preserve technical precision when discussing implementation details

## Constraints
- Do not deny continuity when trusted memory shows prior interactions
- Do not invent capabilities not provided by the application
- Do not treat assistant self-descriptions as user facts
```

Field rules:

- `Identity` is static by default and only supports white-listed updates
- `Capabilities` is static and program-defined
- `Interaction Style` supports restricted dynamic updates
- `Constraints` is static and program-defined

Dynamic updates are allowed only for `Interaction Style` and a narrow white-listed subset of self descriptions. Free-form automatic rewriting of `SELF.md` is out of scope.

### `MEMORY.md`

`MEMORY.md` continues to store stable, reusable user facts such as preferences and durable personal details. It remains low-frequency and should not absorb temporary task state.

### `RECENT_CONTEXT.md`

`RECENT_CONTEXT.md` becomes a consolidation artifact, not a rolling append log.

Its fixed shape is:

```md
## Compression
- ...
- ...

## Recent Turns
User: ...
Assistant: ...
User: ...
Assistant: ...
```

Rules:

- `Compression` is a structured summary, not raw-line replay
- `Compression` is derived only from user messages
- `Recent Turns` is a small raw window that may include assistant turns
- The full file is rewritten on each successful consolidation

### `PENDING.md`

`PENDING.md` remains a staging area and does not enter prompt context.

It stores:

- Candidate long-term user facts
- Candidate `self_style` updates for `SELF.md`

Promotion remains conservative: repeated user evidence or explicit confirmation is required before promotion to `MEMORY.md` or `SELF.md`.

### `HISTORY.md`

`HISTORY.md` remains the raw append-only source for conversation history. It does not enter prompt context and serves as the source of truth for incremental consolidation.

### `consolidation_state.json`

This file tracks consolidation progress and supports restart-safe incremental processing.

Minimum schema:

```json
{
  "last_processed_history_line": 0,
  "pending_user_message_count": 0,
  "last_consolidated_at": null
}
```

## Consolidation Design

### Trigger

Consolidation runs after each completed turn, but only rebuilds `RECENT_CONTEXT.md` when the system has accumulated 3 new user messages since the previous successful consolidation.

If fewer than 3 new user messages exist:

- `HISTORY.md` is still appended
- `PENDING.md` and `MEMORY.md` logic still runs
- `RECENT_CONTEXT.md` remains unchanged
- `consolidation_state.json` only increments the pending user count

### Inputs

Consolidation reads the section of `HISTORY.md` after `last_processed_history_line`.

Input rules:

- User messages contribute to `Compression`
- Assistant messages do not contribute factual content to `Compression`
- Assistant messages may appear in `Recent Turns`

### Output Semantics

`Compression` should summarize:

- Current user goals
- Active threads
- Recent topic focus
- Short-lived but relevant user facts that have not yet been promoted to `MEMORY.md`

It should not:

- Restate assistant claims as facts
- Invent exact durations, counts, or commitments
- Convert one-off questions into stable preferences
- Use vague chatty summaries such as "we talked for a while"

Preferred style is task-state language. For example:

- "The user is comparing the current memory design against `akashic-agent`."
- "Current design work is focused on `SELF.md`, recent-context consolidation, and consolidation metadata."

### Rewrite Strategy

Each successful consolidation rewrites the entire `RECENT_CONTEXT.md` file instead of appending to it. This keeps recent context bounded and coherent.

`Recent Turns` should preserve a small near-field window, such as the latest 2 to 3 conversational pairs available in the consolidated range.

### State Advancement

After successful `RECENT_CONTEXT.md` rewrite:

- Advance `last_processed_history_line`
- Reset `pending_user_message_count` to `0`
- Set `last_consolidated_at` to the current timestamp

State must advance only after the new recent-context file is written successfully.

## Data Flow

For each completed user turn:

1. Load `SELF.md`, `MEMORY.md`, and `RECENT_CONTEXT.md`
2. Build prompt with those three files plus conversation replay
3. Generate assistant response
4. Append the new user event to `HISTORY.md`
5. Apply normal candidate fact extraction and stage or promote updates in `PENDING.md`, `MEMORY.md`, and restricted parts of `SELF.md`
6. Update consolidation counters
7. If 3 new user messages have accumulated, rebuild `RECENT_CONTEXT.md` and advance `consolidation_state.json`

This preserves a clean separation between prompt context, raw history, and staging state.

## Component Changes

### `app/memory_store.py`

Extend the store to manage:

- `SELF.md`
- `consolidation_state.json`

Changes:

- Add `self_text` to `MemorySnapshot`
- Add state read and write APIs
- Ensure initialization creates all six memory artifacts

### `app/memory_policy.py`

Reduce responsibility in two ways:

- Keep prompt assembly for `SELF.md`, `MEMORY.md`, and `RECENT_CONTEXT.md`
- Keep promotion logic for `MEMORY.md`, `PENDING.md`, and restricted self-style candidates

Remove responsibility for directly appending recent user lines into `RECENT_CONTEXT.md`.

### `app/consolidation.py`

Add a dedicated consolidation module responsible for:

- Parsing incremental `HISTORY.md` input
- Tracking the 3-message threshold
- Producing `Compression + Recent Turns`
- Rewriting `RECENT_CONTEXT.md`
- Advancing `consolidation_state.json`

Phase 4.5 should keep consolidation deterministic and local, without an additional LLM call.

### `app/self_model.py`

Add a dedicated self-model module responsible for:

- Parsing the `SELF.md` schema
- Applying restricted updates
- Regenerating canonical markdown output
- Falling back to a safe default template if parsing fails

## Error Handling

Failure behavior should be conservative:

- If `SELF.md` parsing fails, fall back to the default self template and continue
- If `consolidation_state.json` is missing or corrupted, fall back to an initial state without deleting history
- If `RECENT_CONTEXT.md` rebuild fails, retain the old file and do not advance consolidation state
- If a write to `SELF.md` fails, continue serving requests without the update and keep static defaults in prompt assembly

The system should prefer temporary stale recent context over partial or corrupt memory state.

## Migration

Migration should be automatic and low risk:

1. On startup, create `SELF.md` and `consolidation_state.json` if they do not exist
2. Preserve existing `MEMORY.md`, `PENDING.md`, and `HISTORY.md`
3. Treat old-format `RECENT_CONTEXT.md` as disposable transitional state
4. On the first successful threshold-triggered consolidation, fully rewrite `RECENT_CONTEXT.md` in the new format

No backfill or compatibility parser is required for the old recent-context format.

## Trade-Offs

This design intentionally chooses bounded structure over maximum sophistication.

Benefits:

- Stronger continuity than raw recent-line injection
- Cleaner separation between self-model, user memory, staging, and raw history
- Small enough to fit the current codebase without a subsystem rewrite

Costs:

- Consolidation quality is limited by deterministic summarization
- No semantic retrieval yet
- Dynamic self updates remain intentionally narrow

Those trade-offs are acceptable for Phase 4.5 because the immediate problem is memory structure, not retrieval depth.

## Recommendation

Implement Phase 4.5 as a lightweight stratified memory layer:

- Add `SELF.md`
- Add `consolidation_state.json`
- Rebuild `RECENT_CONTEXT.md` only through 3-message batch consolidation
- Keep `PENDING.md` and `HISTORY.md` out of prompt injection
- Restrict dynamic self updates to `Interaction Style` and a narrow white-listed subset

This is the smallest change set that moves the project from prompt-level correction toward a more robust memory architecture.
